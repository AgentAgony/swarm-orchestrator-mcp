## SECTION 2.3: OPTIMISTIC CONCURRENCY CONTROL (OCC)

### 2.3.1 Core Assumption
Data conflicts are statistically rare → more efficient to allow concurrent access without locks

### 2.3.2 Three-Phase OCC Lifecycle for Agents

#### Phase 1: Read (Snapshot)
```
Agent requests resource (e.g., main.py)
Blackboard returns: Content D₀ + Version ID V₀
Version ID types: hash | timestamp | monotonic integer
```

#### Phase 2: Processing (Inference)
```
Agent processes D₀ locally (offline relative to Blackboard)
- No locks held
- Other agents can freely read/write
- Duration: seconds to minutes
Output: Modified state D′
```

#### Phase 3: Validation & Commit
```
Agent submits tuple (D′, V₀) to Blackboard

Blackboard checks: V_curr == V₀?

SUCCESS: V_curr == V₀
├─ Write accepted
├─ D_curr ← D′
└─ Version increments: V₀ → V₁

FAILURE: V_curr ≠ V₀
├─ Another agent modified during processing
├─ Transaction aborted
└─ Agent receives CollisionError
```

### 2.3.3 Starvation Problem in High-Contention Environments

**Issue**: Slow agents repeatedly fail commits due to faster agents updating versions first

**Mitigation Strategies**:

1. **Exponential Backoff**
   - Standard distributed systems pattern
   - Prevents "thundering herd" retry storms

2. **Semantic Merging (Three-Way Merge)**
   ```
   If Agent A modified function X
   And Agent B modified function Y
   → Changes are orthogonal
   → System attempts algorithmic merge
   → Result: V_merged (contains both updates)
   → Saves Agent A inference cost (no re-inference needed)
   ```

### 2.3.4 Atomic File Operations in Python

**Challenge**: OS write syscall does not guarantee atomicity  
**Risk**: System crash mid-write leaves Blackboard corrupted

**Algorithm: Write-Replace Pattern**

```
Step 1: Write to Temporary File
└─ Data written to .tmp file (same filesystem)

Step 2: Flush and Sync
├─ f.flush() → force from OS buffer
└─ os.fsync(f.fileno()) → force to physical disk

Step 3: Atomic Rename
├─ os.replace(src, dst) executed
├─ POSIX compliant + modern Windows
└─ Observers see: complete old file OR complete new file
   (Never partial/corrupted state)

Step 4: Optional Locking (Extreme Safety)
├─ fcntl.flock during version check
└─ Prevents race conditions between check-and-replace
```

## SECTION 2.4: STORAGE ENGINE - SQLite AND WAL MODE

### 2.4.1 Blackboard Metadata Requirements
Metadata stored in structured database (not flat files):
- Agent execution logs
- State tracking
- Reputation scores
- Consensus decisions

### 2.4.2 SQLite Default Bottleneck: Rollback Journal Mode

**Problem**:
```
Writer needs to modify database
├─ Must obtain EXCLUSIVE lock
└─ Blocks ALL readers

Multi-agent system impact:
├─ "Reader" agents (Context Retrievers) need constant access
└─ Severe bottleneck created
```

### 2.4.3 Solution: Write-Ahead Logging (WAL) Mode

**Mechanism**:
```
Changes NOT written to main database immediately
↓
Changes appended to separate .wal file
↓
Enables: Readers read from (main database + WAL)
         While Writer appends to WAL
↓
Result: Effectively implements MVCC (Multiversion Concurrency Control)
├─ Readers see consistent snapshot of past
└─ Writer prepares future state
```

**Implementation in Python**:
```python
PRAGMA journal_mode=WAL;
```

**Concurrency Model**:
- One Writer + Multiple Readers (concurrent)
- Multiple concurrent writers NOT supported
- Solution: "Writer Queue" pattern
  - Single dedicated write thread
  - Serializes all write requests
  - Keeps all reads parallel

## SECTION 2.5: CONFLICT-FREE REPLICATED DATA TYPES (CRDTs)

### 2.5.1 When CRDTs Outperform OCC

**OCC Limitation**: Abort and retry costly for high-latency inference  
**CRDT Advantage**: Real-time collaborative editing without conflicts

**Example Use Case**: Pair Programming
- Agent A writes code
- Agent B writes docstrings
- Simultaneous, conflict-free updates

### 2.5.2 Mathematical Guarantee: Strong Eventual Consistency

```
Condition: All replicas receive same set of operations
Result: All replicas converge to identical state
├─ Regardless of operation ordering
└─ No data loss
```

### 2.5.3 Two CRDT Paradigms

#### Paradigm 1: Operation-Based CRDTs (CmRDT)
```
Mechanism: Agents broadcast operations
├─ Example: insert('a', index=5)
Requirement: Reliable communication channel (Causal Broadcast)
```

#### Paradigm 2: State-Based CRDTs (CvRDT)
```
Mechanism: Agents exchange full local state
Merge: Using semi-lattice merge function (Least Upper Bound)
Advantage: Robust to network partitions
Disadvantage: Bandwidth-intensive
```

### 2.5.4 Sequence CRDTs for Textual Collaboration

**Problem: The Interleaving Problem**
```
Agent A types: "Hello"
Agent B types: "World"
Both at same position

Naive merge result: "HWeolrldo" (incorrect)
```

**Solution: Unique Agent IDs + Deterministic Ordering**

Algorithm: YATA (used in Yjs framework)
```
Treat text as linked list of characters with unique, immutable IDs
├─ Each character: (agent_id, position, char_value)
Ordering rule: Agent A < Agent B
├─ When tie occurs, Site A's characters precede Site B's
Result on all nodes: "HelloWorld" (consistent)
```

### 2.5.5 Python Implementation: pycrdt

**Library**: pycrdt (bindings to Rust implementation Yrs)

**Usage Pattern**:
```
Blackboard maintains: pycrdt.Doc
Agent interaction: Send "update vectors" (binary diffs)
├─ NOT file overwrites
└─ NOT full state transfers

Benefits:
├─ Granular history tracking
├─ "Time Travel" capability
└─ View specific single-agent contributions
    (valuable for Reputation Algorithms in Section 3.3)
```

## SECTION 3.2: WEIGHTED MAJORITY VOTING

### 3.2.1 Limitation of Equal-Vote Systems

**Naive Approach**: All agents vote equally  
**Problem**: Ignores agent capability differences
- GPT-4 (superior reasoning) = same weight as 7B local model
- Suboptimal algorithmic performance

**Solution**: Weighted Majority Voting
```
Each agent assigned scalar weight w_i
Weight reflects: Probability of agent correctness
Result: Better performance than equal voting
```

### 3.2.2 Weight Derivation: Confidence-Based

#### Method 1: Logit-Based Uncertainty
```
Mathematical Foundation:
├─ LLMs predict next-token probability distribution
├─ Perplexity/entropy measures output uncertainty

Formula: Entropy(R) = -Σ_t P(x_t | x_<t) log P(x_t | x_<t)

Interpretation:
├─ Lower entropy = higher confidence
└─ Protocol assigns higher weights to lower-entropy responses
```

#### Method 2: Verbalized Confidence
```
For API-based models (logits unavailable):
├─ Prompt: "Rate confidence: 0.0 to 1.0"
├─ Use rating as weight

Caveat: LLM miscalibration common (overconfidence)
Mitigation: Specific prompting strategies
└─ "Are you sure? Why might you be wrong?"
    Improves calibration accuracy
```

### 3.2.3 Algorithmic Implementation

**Aggregation**:
```
Given:
├─ Votes V = {v₁, v₂, ..., v_n}
├─ Weights W = {w₁, w₂, ..., w_n}

Decision Function:
D = argmax_k Σ_(i=1 to n) w_i · I(v_i = k)

Where I() is indicator function (1 if true, 0 if false)
```

**Empirical Results**:
- Performance improvement on reasoning tasks: >13% vs single-agent
- Effect: Filters stochastic hallucinations effectively

## SECTION 3.3: DEBATE VS. VOTE TRADE-OFF

### 3.3.1 Dual Mechanisms

**Voting**: Acts as filter (accept/reject)  
**Debate**: Acts as refiner (improve reasoning)

### 3.3.2 The Sycophancy Risk

**Problem**: Mode Collapse in Social Space
```
Scenario: Agent 1 makes confident (but wrong) statement
├─ Other agents conform (social pressure)
├─ Cascade of error spreads
└─ Debate degrades into false consensus
```

**Solution: Blind Drafting + Sparse Topology**

#### Protocol Enforcement:

1. **Blind Drafting Phase**
   ```
   Agents generate initial hypothesis independently
   └─ Preserves ensemble diversity
   └─ Prevents early anchoring bias
   ```

2. **Sparse Communication Topology**
   ```
   Instead of: Fully connected graph (everyone ↔ everyone)
   Use: Ring topology or pairs
   
   Effect:
   ├─ Slows consensus propagation
   ├─ Minority opinions survive longer
   └─ Evaluated on merits, not social pressure
   ```

## SECTION 3.4: REPUTATION ALGORITHMS - ELO SYSTEM

### 3.4.1 Adaptive Weighting Over Time

**Goal**: Make weights w_i dynamic and self-improving

**Algorithm**: Elo Rating (familiar from Chess)

### 3.4.2 Mechanism

```
When result validated (e.g., unit test passes):
├─ Agents voting CORRECT answer: gain rating points
└─ Agents voting WRONG answer: lose rating points

Convergence: Over time, weights reflect true agent capabilities
```

### 3.4.3 Domain-Specific Ratings

```
Maintain separate Elo ratings per domain:
├─ R_Python
├─ R_SQL
├─ R_CreativeWriting
└─ ... per specialty

Effect: System learns to route tasks optimally
├─ "Trust Agent A for coding"
├─ "Trust Agent B for prose"
└─ Voting weights dynamically optimize
```

## SECTION 5.3: RETRIEVAL ALGORITHMS

### 5.3.1 Personalized PageRank (PPR)

**Analogy**: Mimics human associative memory

**Algorithm Steps**:

1. **Seed Identification**
   ```
   User query mapped to graph nodes
   Example: "Fix the bug in process_payment"
   └─ Seed node: function "process_payment"
   ```

2. **Random Walk Simulation**
   ```
   Imaginary walker starts at seed node
   └─ Traverses edges probabilistically
   ```

3. **Spread Activation**
   ```
   Walker traverses edges (calls, imports)
   └─ Nodes structurally related to seed receive probability mass
   
   Example: validate_currency (called by process_payment)
   ├─ Structurally connected
   ├─ Selected for context even without keyword match
   └─ Captures hidden multi-hop relationships
   ```

4. **Context Selection**
   ```
   Top K nodes by PPR score
   └─ Retrieved and added to context window
   ```

**Performance**: Significantly outperforms vector-only retrieval on multi-hop reasoning

### 5.3.2 HippoRAG Advantage

**Finding**: Graph-based retrieval captures hidden connections vector methods miss  
**Result**: Better performance on reasoning tasks requiring structural understanding

## SECTION 6.2: TEXTRANK FOR SUMMARIZATION

### 6.2.1 Algorithm Overview

**GraphRAG → TextRank for further compression**

```
Step 1: Build Graph

├─ Select top N% of ranked sentences

```

**Typical Compression**: 90% reduction (retain top 10% of sentences)  
**Preservation**: Key information + debate flow maintained

## SECTION 6.3: SELECTIVE CONTEXT - ENTROPY-BASED FILTERING

### 6.3.1 Information-Theoretic Approach

**Foundation**: Information Theory  
**Unit**: Token-level compression (more granular than TextRank)

### 6.3.2 Self-Information Calculation

```
Self-Information (Surprisal):
I(w_i) = -log P(w_i | w_<i)

Interpretation:
├─ High I(w_i): Surprising, informative token
└─ Low I(w_i): Predictable, redundant token
```

### 6.3.3 Filtering Algorithm

```
Step 1: Calculate Self-Information
├─ Use lightweight LM (GPT-2) to compute perplexity
└─ Calculate I(w_i) for every token

Step 2: Identify Redundant Tokens
├─ Low self-information: "the", "is", "a"
└─ Mark for pruning

Step 3: Prune Low-Information Tokens
├─ Remove from context
└─ Reconstruct prompt

Result: 40-50% length reduction, negligible LLM performance loss
```

**Impact**: Significant latency and cost reduction

## SECTION 7.3: SYMBOLIC EXECUTION WITH CROSSHAIR

### 7.3.1 Testing vs. Verification

**Testing**: Checks specific inputs (sampling)  
**Verification**: Checks ALL possible inputs (exhaustive, mathematical)

### 7.3.2 CrossHair Tool

**Tool**: CrossHair (Python verification)  
**Backend**: Z3 SMT Solver (Satisfiability Modulo Theories)

### 7.3.3 Mechanism

```
Traditional Testing:
├─ x = 5 (concrete value)
└─ Run code, check result

Symbolic Execution (CrossHair):
├─ x = SymbolicInt (all possible integers)
├─ Mathematically explore execution paths
└─ Find conditions violating contracts
```

### 7.3.4 Workflow Integration

```
Verification Agent runs CrossHair on agent-generated code
├─ CrossHair finds counter-example
├─ Example: "Function fails when list is empty"
├─ This input added to Blackboard as test case
└─ Coder Agent forced to patch edge case

Result: Deterministic proof of correctness (within scope)
```

## SECTION 7.4: SPECTRUM-BASED FAULT LOCALIZATION (SBFL)

### 7.4.1 Problem: Where's the Bug?

**Challenge**: When tests fail, agents struggle to locate bug  
**Solution**: SBFL automates diagnosis using execution traces

### 7.4.2 The Ochiai Algorithm

```
Step 1: Run Full Test Suite
├─ Record coverage spectrum
└─ Which code lines executed by:
   ├─ Passing tests (P)
   └─ Failing tests (F)

Step 2: Calculate Suspiciousness Score
S(l) = failed(l) / √(total_failed · (failed(l) + passed(l)))

Interpretation:
├─ S ≈ 1.0: Line strongly correlated with failure
│  └─ Executed mostly in failing tests
├─ S ≈ 0.0: Line safe
└─ Ranking: Lines by suspiciousness score
```

### 7.4.3 Automated Debugging Prompt

```
Protocol generates specific Debugging Prompt:

"The tests failed. The Ochiai algorithm analysis:
├─ Line 42: Suspiciousness = 0.98
├─ Line 45: Suspiciousness = 0.96
└─ These are the most likely bug locations.
   Analyze these lines first."

Effect: Algorithmic attention mechanism
└─ Drastically improves LLM agent fix rate
```

### 7.4.4 APR Loop: Automated Program Repair

```
Cycle:
├─ Agent generates code
├─ Tests fail
├─ SBFL identifies suspects (Lines 42, 45)
├─ Agent receives targeted debugging prompt
├─ Agent analyzes high-suspicion lines
├─ Agent generates fix
└─ Tests run again

Convergence: Tight loop toward correctness
```
